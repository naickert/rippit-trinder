Provably fair

See exactly how every card is decided.

Every pack outcome is locked in at purchase time with a cryptographic commitment. The server cannot change the result after you buy, and you can independently recalculate the same roll, rarity, and selection path from the revealed proof.

How the flow works

A deterministic HMAC-SHA256 calculation over published rarity weights. Same inputs always produce the same result.

1. Commit

When you buy, the server generates a hidden serverSeed and shows only its SHA-256 hash before the rip.

2. Derive

For each slot we compute HMAC(serverSeed, clientSeed:nonce:slotIndex).

3. Choose rarity

The first 32 bits become a roll in [0,1) placed into the cumulative rarity bins.

4. Choose card

A second HMAC with a :card:N suffix is rejection-sampled — uniform, no modulo bias.

message = clientSeed + ":" + nonce + ":" + slotIndex hmac = HMAC_SHA256(serverSeed, message) roll = first32Bits(hmac) / 2^32 rarity = cumulativeBin(roll, rarityWeights)
